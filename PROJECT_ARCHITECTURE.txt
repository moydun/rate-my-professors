Rate My Professors Kyrgyzstan - Architecture
============================================

Document status: implementation architecture baseline
Related requirements: PROJECT_REQUIREMENTS.txt
Current implementation branch: Phase 2 domain models

This file is the source of truth for technical architecture. Product goals,
user-facing MVP scope, moderation policy, testing expectations, and delivery
phases belong in PROJECT_REQUIREMENTS.txt. Avoid duplicating architecture details
there; reference this file instead.


1. Architectural Position
-------------------------

1.1 System style

The project is a modular Django monolith.

Required properties:

- One deployable Django application.
- Server-rendered HTML through Django Templates.
- Django Admin as the MVP data-management and moderation surface.
- Django ORM as the only application database access layer.
- PostgreSQL as the production database.
- SQLite allowed only for local development and tests.
- HTMX allowed only for progressive enhancement, not as an SPA replacement.

Rejected for MVP:

- Separate React/Vue/Angular frontend.
- Public REST API.
- Microservices.
- External search service.
- Machine-learning moderation.
- Professor self-service dashboards.
- University representative dashboards.

1.2 Why a Django monolith

The product needs forms, lists, search, detail pages, authentication, moderation,
and admin workflows. A separate frontend/backend split would add API design,
CORS, duplicated validation, token/session complexity, and more deployment
moving parts before the product has enough traffic or workflow complexity to
justify it.

Django already provides the right primitives for this phase:

- auth and permissions;
- CSRF/session security;
- forms and validation;
- templates;
- ORM and migrations;
- admin;
- test client and model tests.

1.3 Current implementation status

Implemented:

- Valid Django project package: `rmp`.
- Local settings with environment-based production guards.
- Apps registered in settings: `accounts`, `universities`, `professors`,
  `reviews`.
- Initial models and migrations for universities, departments, professors,
  courses, reviews, and review reports.
- Django Admin registration for domain models.
- Review public querysets and rating summaries.
- Bulk admin moderation actions.
- Focused review/model tests.

Not implemented yet:

- Public pages.
- Search views.
- Authentication views.
- Review submission forms.
- Report submission forms.
- Production PostgreSQL configuration.
- Deployment files.


2. Runtime Topology
-------------------

2.1 Local development

Local development uses:

- Python virtual environment.
- Django development server.
- SQLite database file generated by migrations.
- Django static-file behavior for development.

Local artifacts must not be committed:

- `db.sqlite3`
- `__pycache__/`
- `.venv/`
- `staticfiles/`
- `.env`

2.2 Default production topology

The default production topology is a small VPS:

Internet
    -> Nginx
    -> Gunicorn
    -> Django application
    -> PostgreSQL

Responsibilities:

- Nginx terminates HTTP/HTTPS traffic and serves static files.
- Gunicorn runs the Django WSGI application.
- Django handles request routing, auth, business logic, templates, admin, and
  ORM access.
- PostgreSQL stores all production data.

2.3 Static files

Default VPS decision:

- Run `python manage.py collectstatic` during deployment.
- Configure `STATIC_ROOT`.
- Configure Nginx to serve `/static/` from `STATIC_ROOT`.
- Do not use Django development static serving in production.
- Do not use WhiteNoise in the default VPS deployment.

Alternative:

- WhiteNoise may be used for a PaaS/container deployment where Nginx is not
  serving static files.
- Choosing WhiteNoise must be an explicit deployment decision.

Media uploads:

- User-uploaded media is deferred.
- If media is added later, do not serve it through WhiteNoise by default.
- Choose Nginx media serving or object storage explicitly.

2.4 Process management

Recommended production process management:

- systemd service for Gunicorn.
- Gunicorn bound to localhost or a Unix socket.
- Nginx as the public reverse proxy.
- Environment variables loaded from systemd EnvironmentFile or equivalent.

2.5 Required production environment variables

- `DJANGO_SECRET_KEY`
- `DJANGO_DEBUG=False`
- `DJANGO_ALLOWED_HOSTS`
- `DATABASE_URL` or explicit PostgreSQL settings
- Email backend settings before public password reset is enabled


3. Configuration Architecture
-----------------------------

3.1 Django package

The Django project package is `rmp`.

Required references:

- `DJANGO_SETTINGS_MODULE = rmp.settings`
- `ROOT_URLCONF = rmp.urls`
- `WSGI_APPLICATION = rmp.wsgi.application`
- `ASGI_APPLICATION = rmp.asgi.application`

The package name must remain importable Python. Do not use hyphens in package
names.

3.2 Settings policy

Development:

- Safe defaults may exist for local development.
- `DEBUG` defaults to true locally.
- SQLite is acceptable locally.

Production:

- `DJANGO_SECRET_KEY` is required.
- `DJANGO_DEBUG=False` is required.
- `DJANGO_ALLOWED_HOSTS` is required.
- Secrets must come from environment variables.
- Production database credentials must not be committed.

3.3 Timezone and localization

Current project timezone:

- `Asia/Bishkek`

Language policy:

- MVP public UI target: Russian.
- Data must support Unicode Russian, Kyrgyz, and English content.
- Full Kyrgyz UI localization is deferred, but the architecture must not block
  Django i18n.


4. App Boundaries
-----------------

4.1 `rmp`

Owns:

- settings;
- root URLs;
- ASGI/WSGI entrypoints.

Must not own:

- domain models;
- review business rules;
- public page logic beyond temporary bootstrapping.

4.2 `accounts`

Owns:

- registration;
- login/logout;
- password reset wiring;
- profile pages;
- account-related policies.

Current decision:

- Use Django's built-in User model for MVP unless a custom user model is chosen
  before user-related production migrations are finalized.

Reasoning:

- The external SQLite schema proposes a custom `users` table with UUID ids,
  email login, roles, verification, avatar, and timestamps.
- In this Django app, those concerns can be handled through Django User, Groups,
  Permissions, and optional profile fields later.
- A custom user model is expensive to introduce after migrations; if the team
  wants email-as-login or UUID users, decide before public production data.

4.3 `universities`

Owns:

- universities;
- departments/faculties;
- public active filtering for university metadata.

Terminology:

- The external schema uses `schools`.
- This project uses `universities` because the product is Kyrgyzstan-focused and
  university/faculty/department terminology is clearer locally.

4.4 `professors`

Owns:

- professor profiles;
- professor-department relationships;
- courses;
- professor-course relationships;
- professor rating summary entry point.

Does not own:

- review storage;
- moderation decisions;
- report workflows.

4.5 `reviews`

Owns:

- review records;
- review status values;
- public review querysets;
- rating aggregation;
- duplicate review constraints;
- report records;
- report resolution;
- admin moderation actions.

Rule:

- Public pages must use `Review.objects.public()` or a queryset derived from it.
- Avoid instance-level visibility checks that query per object and cause N+1
  query patterns.

4.6 Future `core`

Allowed only when needed for:

- static pages;
- shared layout/context helpers;
- small cross-project utilities.

It must not become a dumping ground for domain behavior.


5. Domain Model
---------------

5.1 Overview

Logical relationships:

- University has many Departments.
- Department belongs to one University.
- Professor belongs to one or more Departments.
- Course belongs to one University and one Department.
- Course can be linked to many Professors.
- Professor can teach many Courses.
- Review belongs to one Professor.
- Review optionally belongs to one Course.
- Review belongs to one Author.
- ReviewReport belongs to one Review and one Reporter.

Mermaid overview:

```mermaid
erDiagram
    UNIVERSITY ||--o{ DEPARTMENT : contains
    UNIVERSITY ||--o{ COURSE : offers
    DEPARTMENT ||--o{ COURSE : owns
    PROFESSOR }o--o{ DEPARTMENT : belongs_to
    PROFESSOR }o--o{ COURSE : teaches
    PROFESSOR ||--o{ REVIEW : receives
    COURSE ||--o{ REVIEW : contextualizes
    USER ||--o{ REVIEW : writes
    REVIEW ||--o{ REVIEW_REPORT : receives
    USER ||--o{ REVIEW_REPORT : submits
```

5.2 University

Current Django model:

- `id`
- `name`
- `slug`
- `city`
- `website_url`
- `description`
- `is_active`
- `created_at`
- `updated_at`

Constraints:

- `slug` is unique globally.
- `name` plus `city` is unique.

Public queryset:

- `University.objects.public()` returns active universities only.

External schema comparison:

- External `schools` table includes `state`, `country`, `type`,
  `avg_overall_rating`, and `num_school_ratings`.
- For Kyrgyzstan MVP, `city` is required and `country` is implicit.
- Cached school aggregates are deferred.
- `state` and school `type` are not required for MVP.

Deferred fields:

- country;
- type;
- cached average school rating;
- cached rating count.

5.3 Department

Current Django model:

- `id`
- `university`
- `name`
- `slug`
- `is_active`
- `created_at`
- `updated_at`

Constraints:

- `university` plus `slug` is unique.
- `university` plus `name` is unique.

Public queryset:

- `Department.objects.public()` returns active departments whose universities
  are active.

External schema comparison:

- External `departments` table maps directly, except it uses `school_id`.
- This project uses `university_id`.

5.4 Professor

Current Django model:

- `id`
- `first_name`
- `last_name`
- `middle_name`
- `display_name`
- `slug`
- `departments`
- `bio`
- `is_active`
- `created_at`
- `updated_at`

Relationships:

- Many-to-many with Department.

Public queryset:

- `Professor.objects.public()` returns active professors with at least one
  active department connected to an active university.

External schema comparison:

- External `professors` table includes `school_id`,
  `primary_department_id`, `title`, `profile_image_url`, `is_claimed`,
  `claimed_by_user_id`, cached averages, and rating counts.
- This project keeps department membership as the source of university
  affiliation instead of storing direct `school_id`.
- Primary department is deferred; it can be added later if profile display needs
  one canonical department.
- Professor claim workflow is deferred.
- Profile images are deferred.
- Cached professor aggregates are deferred.

Deferred fields/features:

- primary department;
- title;
- profile image;
- claimed professor workflow;
- cached average rating;
- cached difficulty rating;
- cached would-take-again percentage;
- cached rating count.

5.5 Course

Current Django model:

- `id`
- `university`
- `department`
- `professors`
- `code`
- `name`
- `is_active`
- `created_at`
- `updated_at`

Relationships:

- Belongs to one University.
- Belongs to one Department.
- Many-to-many with Professor.

Constraints:

- Non-empty `code` is unique per University.

Reasoning:

- The external schema uses `UNIQUE(department_id, code)`.
- Current project uses `UNIQUE(university, code)` for non-empty codes so the
  same official course code cannot accidentally be duplicated across departments
  in one university.
- If Kyrgyzstan universities reuse course codes across faculties in practice,
  change the constraint before production data.

Public queryset:

- `Course.objects.public()` returns active courses whose university and
  department are active.

5.6 Review

Current Django model:

- `id`
- `professor`
- `course`
- `author`
- `academic_year`
- `term`
- `rating`
- `difficulty`
- `would_take_again`
- `comment`
- `status`
- `moderation_note`
- `moderated_by`
- `moderated_at`
- `created_at`
- `updated_at`
- `published_at`

Status values:

- `pending`
- `published`
- `hidden`
- `rejected`

MVP status policy:

- New reviews start as `pending`.
- Only `published` reviews are visible publicly.
- Only `published` reviews affect aggregates.
- `hidden` and `rejected` reviews remain stored for moderation/audit.

Public queryset:

- `Review.objects.public()` filters to published reviews connected to active
  professors, active departments, and active universities.

Duplicate-review constraints:

- One active review per author, professor, course, and academic year when course
  is known.
- One active review per author, professor, and academic year when course is
  unknown.
- Active means `pending` or `published`.
- Hidden/rejected reviews do not block replacement reviews.

External schema comparison:

- External schema uses `overall_rating` and `difficulty_rating`; current model
  uses `rating` and `difficulty`.
- External schema defaults reviews to `published`; current MVP defaults to
  `pending` to protect trust and moderation.
- External schema includes `school_id` directly on reviews. Current model derives
  university through professor/course relationships to avoid denormalized
  inconsistency in MVP.
- External schema includes `grade_received`, `attendance_mandatory`,
  `textbook_required`, `is_online_class`, `helpful_count`,
  `not_helpful_count`, and review tags. These are deferred unless the first UI
  explicitly needs them.
- External status values `flagged` and `removed` map conceptually to report
  state plus `hidden`/`rejected`; current project keeps review status and report
  status separate.

Deferred review fields/features:

- grade received;
- attendance mandatory;
- textbook required;
- online class flag;
- helpful/not-helpful votes;
- tags;
- cached vote counters;
- denormalized university id on Review.

5.7 ReviewReport

Current Django model:

- `id`
- `review`
- `reporter`
- `reason`
- `details`
- `status`
- `resolved_by`
- `resolved_at`
- `created_at`

Reason values:

- `spam`
- `abusive_content`
- `personal_data`
- `false_or_misleading`
- `duplicate`
- `other`

Status values:

- `open`
- `resolved`
- `rejected`

Constraints:

- One report per review per reporter.

External schema comparison:

- External reason values include `offensive`, `false_info`, and
  `wrong_professor`.
- Current reasons are broader but cover the same moderation categories.
- `wrong_professor` may be added later if report volume shows it is common.
- External statuses `pending`, `reviewed`, `dismissed`, `action_taken` map to
  current `open`, `resolved`, and `rejected`.

5.8 Deferred tables from external schema

The pasted SQLite schema includes several useful future features. They are not
MVP requirements yet.

Tags:

- External table: `tags`
- External join: `review_tags`
- Current decision: deferred.
- Reason: structured review fields and moderation are higher priority.

Review votes:

- External table: `review_votes`
- Current decision: deferred.
- Reason: votes add abuse surface and require anti-gaming rules.

Saved professors:

- External table: `saved_professors`
- Current decision: deferred.
- Reason: bookmark UX is useful later but not needed for the core review
  platform.

Professor claims:

- External fields: `is_claimed`, `claimed_by_user_id`
- Current decision: deferred.
- Reason: self-service professor workflows are explicitly outside MVP.

Cached aggregates:

- External fields: school/professor average ratings and counts.
- Current decision: compute from published reviews in MVP.
- Reason: correctness is easier while traffic is low.
- Future rule: if cached aggregates are introduced, updates must be centralized
  and tested against review publish/hide/reject transitions.


6. Database and Persistence
---------------------------

6.1 Database choice

Local:

- SQLite is allowed.
- SQLite database files are generated locally and ignored by git.

Production:

- PostgreSQL is required.

Reasoning:

- better concurrent writes;
- better indexing options;
- full-text and trigram search options;
- production-like behavior for deployment.

6.2 Identifier strategy

Current implementation:

- Django default `BigAutoField` ids.

External schema:

- UUID stored as text.

Current decision:

- Keep `BigAutoField` for MVP.
- UUID migration is not needed unless public API exposure, distributed imports,
  or multi-system data sync becomes a real requirement.

6.3 Derived and cached data

MVP:

- Rating summaries are computed from `published` Review rows.
- Cached aggregate columns are not stored.

Future cached aggregate requirements:

- update when a review is published;
- update when a review is hidden;
- update when a review is rejected;
- update when rating fields change before publication;
- exclude pending/hidden/rejected;
- include tests for every transition.

6.4 Indexing strategy

Current priority:

- rely on constraints and Django-created relation indexes first;
- add explicit indexes only when query patterns exist.

Likely future indexes:

- University name/city search.
- Professor first/last/display name search.
- Course code/name search.
- Review status plus professor.
- Review status plus created/published timestamp.
- Report status plus created timestamp.

PostgreSQL future search:

- `pg_trgm` for fuzzy professor/university/course search; or
- PostgreSQL full-text search if query semantics need ranking.

Do not add Elasticsearch/Meilisearch in MVP.

6.5 Deletion behavior

Current policy:

- User-generated review content should not be hard-deleted by default.
- Moderation uses statuses.
- Hard deletion is reserved for legal, privacy, or operational needs.

Database behavior:

- Professor deletion cascades reviews in the current model.
- In production workflows, admins should prefer deactivation/hiding over
  deletion for public domain objects.

Future consideration:

- Add stricter admin policy or soft-delete behavior if real production data
  requires stronger audit guarantees.


7. Review and Moderation Architecture
-------------------------------------

7.1 Review lifecycle

Default lifecycle:

pending
    -> published
    -> hidden

Alternative terminal path:

pending
    -> rejected

Rules:

- New user submissions start `pending`.
- Moderators/admins publish valid reviews.
- Moderators/admins hide already-published content when it violates policy or
  needs removal from public pages.
- Moderators/admins reject pending content that should not be published.

7.2 Public visibility

Public review visibility is queryset-based:

- use `Review.objects.public()`;
- do not use an instance property that queries related departments per object;
- avoid N+1 query patterns in lists and templates.

7.3 Rating aggregation

Professor rating summaries use:

- published reviews only;
- average rating;
- average difficulty;
- count of published reviews;
- percentage of published reviews where `would_take_again=True`.

Current implementation:

- `Professor.rating_summary()`
- `ReviewQuerySet.published_summary()`

7.4 Reports

Report rules:

- Reports target reviews.
- Reporter must be a user.
- One reporter can report a review once.
- Reports have structured reason values and optional details.
- Resolution records resolver and timestamp.

7.5 Admin actions

Current admin actions:

- publish reviews;
- hide reviews;
- reject reviews;
- resolve reports.

Implementation rule:

- Use bulk `queryset.update(...)` where no per-object side effects are needed.
- Use one shared timestamp per batch action.
- Preserve existing `published_at` when republishing already-published reviews.


8. Search Architecture
----------------------

8.1 MVP search

Use Django ORM queries.

Targets:

- professor names;
- university names;
- course codes;
- course names.

MVP matching:

- exact matches first where practical;
- partial matches with `icontains`;
- pagination for result lists;
- minimum query length for live search if HTMX suggestions are added.

8.2 Future search

If MVP search is insufficient:

- add PostgreSQL trigram search;
- add PostgreSQL full-text search;
- add ranking rules.

External search services remain deferred.


9. View and Template Architecture
---------------------------------

9.1 Server-rendered first

Public pages should be Django views rendering Django Templates.

Required future pages:

- home/search page;
- search results;
- university detail;
- professor detail;
- review submission;
- profile page;
- static policy pages.

9.2 HTMX usage

Allowed:

- live search suggestions;
- review filtering;
- report modal submission;
- load-more reviews.

Not allowed:

- hiding backend complexity behind client state;
- making core review submission require SPA behavior;
- duplicating validation in JavaScript as the source of truth.


10. Security Architecture
-------------------------

10.1 Django defaults

Keep enabled:

- CSRF protection;
- Django auth password hashing;
- session middleware;
- clickjacking protection;
- template escaping.

10.2 Production security

Required before public launch:

- `DEBUG=False`;
- explicit `ALLOWED_HOSTS`;
- secret key from environment;
- database credentials from environment;
- HTTPS;
- secure cookies after HTTPS is configured;
- restricted staff/admin access.

10.3 User-generated content

MVP:

- plain text comments only;
- templates escape user text;
- no rich text;
- no user-uploaded media.

Future rich text/media:

- sanitize rich text server-side;
- validate uploads at the boundary;
- store media outside the database;
- explicitly choose media serving/storage architecture.


11. Testing Architecture
------------------------

11.1 Current test focus

Implemented tests cover:

- review default pending status;
- rating/comment validation;
- published-only aggregation;
- public queryset behavior;
- duplicate active review policy;
- report resolution;
- admin bulk publish;
- admin bulk report resolution.

11.2 Required future tests

When views are added:

- anonymous users cannot submit reviews;
- authenticated users can submit reviews;
- review submissions become pending;
- hidden/rejected reviews are not publicly visible;
- search returns expected university/professor/course matches;
- report submission creates a report record;
- non-staff users cannot access moderation surfaces.


12. Migration and Evolution Rules
---------------------------------

12.1 Before public production data

Schema can still evolve quickly, but migrations must remain runnable from an
empty database.

12.2 After public production data

Changes must preserve data:

- no destructive migrations without explicit backup plan;
- no field renames without migration strategy;
- no status-value changes without data migration;
- no custom user model switch without a dedicated migration plan.

12.3 When using external schema ideas

The pasted SQLite schema is useful as a product/DB reference, but it must be
adapted through Django models and current MVP decisions.

Decision matrix:

External idea                         Current decision
------------------------------------  -----------------------------------------
`schools` table                       Use `universities`.
UUID text ids                         Keep Django `BigAutoField` for MVP.
Custom users table                    Use Django User for MVP.
User roles column                     Use Django staff/groups/permissions first.
Email verification                    Recommended later, not required for first
                                      private/test deployment.
Cached school/professor aggregates    Defer; compute from published reviews.
Professor primary department          Defer until profile display requires it.
Professor claim workflow              Deferred non-goal for MVP.
Profile images                        Deferred.
Course department-code uniqueness     Current code uses non-empty code unique per
                                      university; revisit before production if
                                      real data needs department scope.
Review default `published`            Reject; MVP default is `pending`.
Review `flagged`/`removed` statuses   Use reports plus `hidden`/`rejected`.
Grade/attendance/textbook fields      Deferred unless first review UI needs them.
Review tags                           Deferred.
Helpful/not-helpful votes             Deferred.
Saved professors                      Deferred.
Direct `school_id` on reviews         Defer denormalization.


13. Architecture Acceptance Criteria
------------------------------------

The architecture baseline is acceptable for Phase 2 when:

- Django imports `rmp.settings` successfully.
- Migrations run from an empty database.
- Domain models are registered in Django Admin.
- Public review querysets exclude non-public content.
- Rating summaries count only published reviews.
- Review duplicate constraints protect active review duplication.
- Admin moderation actions update audit fields.
- Tests cover review status, aggregation, public visibility, duplicate policy,
  reports, and admin bulk actions.

Current branch satisfies these criteria through the Phase 2 implementation.
